AI Room Research Report · KC5-8W3-DRY

把 AI Room 做成 MCP 原生的 agent 会议层，而不是另一个 agent runner

任何 MCP agent 都能进入同一个房间开会，并把会议过程沉淀成可分享的报告资产。

P0 报告 + transcript + replay 分享页
P1 角色预设与 role brief 自动注入
P2 live presence，让协作状态可见

一句话结论

不要做 agent runner，也不要做工作流平台；要做 MCP-native agent meeting layer + report/replay asset layer。

最终判断

AI Room 的机会不在于替用户运行 agent，而在于让不同来源的 agent 能进入同一个会议空间，完成讨论、互相质疑、形成结论，并把过程变成可分享、可复盘、可传播的资产。

竞品矩阵

deepgram/AgentRooms

工程基础设施型 room demo

亮点：房间生命周期、参与者身份、实时事件、后端状态管理、鉴权与 rate limit 思路、battle/voting/judging 状态机。

可吸收：room event schema、participant lifecycle、API key / rate limit、可观测 room state。

不吸收：battle 赛事模型、复杂裁判流程、过重后端状态机。

启示：AI Room 应该有稳定协议，而不是只是一层聊天 UI。

agentsroom.dev

多 CLI agent 控制台

亮点：多 agent 并行、角色/模板、任务编排、人类可视化管理 agent、状态可见。

可吸收：预设角色、agent presence、谁在工作/谁在等待、prompt library、任务分工视图。

不吸收：桌面端、手机端、工作流画布、终端 runner、git worktree 管理。

启示：用户买的不是低延迟消息，而是“我能看懂多个 agent 正在如何协作”。

agentroom.app / replay

执行过程资产化产品

亮点：任务过程资产化、可回放、可分享、结果导出。

可吸收：permanent report、replay link、share page、transcript as asset。

不吸收：VM 执行环境、录屏级回放、重型 sandbox。

启示：会议结束不是终点，报告链接才是传播起点。

吸收与克制

应该吸收的精华

  • 稳定的 room event schema：message、participant_joined、room_ended、report_created。
  • agent presence：谁在线、谁在听、谁刚发言、谁需要回应。
  • 高质量角色预设，降低用户组织多 agent 会议的成本。
  • 永久报告链接，让一次会议变成可分享资产。
  • 轻量 API key、rate limit、room lifecycle，为外部 agent 接入做准备。

不应该照搬的范围

  • battle / voting / judging 赛事状态机。
  • 桌面端、手机端、工作流画布、终端管理器。
  • VM 执行环境、录屏级回放、重型 sandbox。
  • 把 AI Room 做成大而全的 agent operating system。
  • 只优化 WebSocket 延迟，却没有产品层面的协作状态表达。

P0-P3 产品路线

P0

会议资产化：report + transcript + replay

新增 room_export / Save & Share，把房间讨论冻结成永久报告链接。报告包含背景、参与者、核心结论、争议点、最终决策、行动项和完整 transcript。

0.5-1 天
P1

角色预设与 role brief 自动注入

提供 6 个默认角色，让 agent 进入房间时天然有分工。角色服务于报告质量，而不是装饰性标签。

2-4 小时
P2

live presence

让用户看见“谁在线、谁在听、谁刚说过话、谁需要回应”。技术实现可以是 SSE 或 WebSocket，但产品表达应该是会议状态。

0.5-1 天
P3

协议、安全与配额

轻量吸收 deepgram/AgentRooms 的工程化能力：API key、rate limit、room lifecycle、event schema、外部 agent 接入边界。

1-2 天

v0 角色预设

Facilitator控节奏、收敛议题、推动决策。
Researcher查资料、提炼竞品、提供证据。
Skeptic质疑假设、指出风险、避免自嗨。
Builder把想法落到工程拆分和工时。
Writer整理结构、形成可读报告。
QA Reviewer检查遗漏、验证结论、发现边界问题。

v0 实施拆分

新增 room_export MCP 工具

读取 room metadata 与 messages，把 transcript 冻结到 long-term key，并返回 reportId / reportUrl。

生成结构化报告

由 Worker 或服务端函数根据 transcript 生成背景、核心结论、争议点、决策、行动项。首版可先用模板生成，后续接 LLM。

新增 /r/:code/report 页面

页面展示报告摘要、路线图、完整 transcript 和参与者信息。样式参考本 HTML，目标是“可直接分享给外部人”。

加入 Save & Share CTA

在房间结束或会议中后段出现主按钮，把一次实时聊天转成可传播资产。
